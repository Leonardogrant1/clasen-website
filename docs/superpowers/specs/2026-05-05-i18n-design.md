# i18n Localization — Design Spec

**Date:** 2026-05-05
**Status:** Approved

## Overview

Add DE/EN localization to the Clasen Immos Next.js 16 app using **next-intl**. German is the default locale (no URL prefix). English lives at `/en/...`. A small fixed language picker sits bottom-right on all pages.

---

## Routing

- **German (default):** `/`, `/wohnen-und-leben`, `/clasen`, `/contact`, etc. — no prefix
- **English:** `/en`, `/en/wohnen-und-leben`, `/en/clasen`, `/en/contact`, etc.
- Implemented via next-intl middleware with `localePrefix: 'as-needed'`
- Middleware file: `middleware.ts` at project root

---

## File Structure Changes

```
messages/
  de.json          ← all German strings
  en.json          ← all English strings

app/
  [locale]/        ← NEW dynamic segment (wraps all existing routes)
    layout.tsx     ← NextIntlClientProvider + html lang attribute
    page.tsx
    wohnen-und-leben/page.tsx
    clasen/page.tsx
    contact/page.tsx
    commercial/page.tsx
    living/page.tsx
    login/page.tsx

i18n/
  routing.ts       ← locales config + createNavigation helpers
  request.ts       ← getRequestConfig for server components

middleware.ts      ← next-intl createMiddleware
```

Existing `app/*.tsx` pages and the root `app/layout.tsx` are replaced by the `[locale]/` equivalents.

---

## Translation Keys

All user-visible strings are extracted into `messages/de.json` and `messages/en.json`. Structure:

```json
{
  "nav": {
    "kapital": "KAPITAL",
    "wohnenUndLeben": "WOHNEN & LEBEN",
    "clasen": "Clasen",
    "contact": "Kontakt",
    "login": "Mandanten Login"
  },
  "hero": {
    "label": "Ihre Vision",
    "items": [
      { "title": "Aufs richtige \nPferd setzen.", "subtitle": "..." },
      { "title": "Vor-Sorge statt Nach-Sicht", "subtitle": "..." }
    ]
  },
  "testimonials": {
    "items": [
      { "quote": "...", "type": "Zufriedener Käufer", ... },
      ...
    ]
  },
  "footer": {
    "rights": "All rights reserved"
  }
  // ... one namespace per component
}
```

`heroMapping` and `testimonials` in `constants.ts` are converted to locale-aware translation keys rather than static data. Components receive translated strings from `useTranslations()`. `HeroSection` continues to pick a random hero index via `Math.random()` — it reads that index's translated title/subtitle from the messages array rather than from `constants.ts` directly.

## Language Picker Component

**File:** `components/LanguagePicker.tsx`

- Fixed position: `bottom-6 right-6` (always visible, overlays content)
- Visual: small pill `bg-white/10 backdrop-blur border border-white/20 rounded-full px-3 py-1.5`
- Content: `DE | EN` — active locale in accent color `#C9A84C`, inactive in `text-foreground/40`
- On click: uses next-intl's `usePathname` + `useRouter` to navigate to the same route in the other locale (e.g. `/wohnen-und-leben` ↔ `/en/wohnen-und-leben`)
- Rendered in the `[locale]/layout.tsx` so it appears on every page
- `"use client"`

---

## Metadata

Each page exports locale-aware `generateMetadata`. Title and description strings live in the messages files under a `metadata` namespace per page.

---

## What Is NOT Translated

- Property names, locations, and client names in testimonials (proper nouns, stay as-is in both locales)
- The logo, images, and videos (language-neutral)
- The `/login` page internals (out of scope for this iteration)

---

## Open Questions / Deferred

- `/en/` page URL slugs: `wohnen-und-leben` stays as German slug in English URLs for now (no URL slug translation). Can be added later.
- Auto-detection of browser language: not implemented. Users start on DE and switch manually.
